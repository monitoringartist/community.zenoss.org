threshold of zenperfsnmp cycle time exceeded

In the event console, I regularly receive the following critical alert:

threshold of zenperfsnmp cycle time exceeded: current value 243.93


I had a couple questions regarding why I'm getting this alert. Firstly, if I go to /Monitors/localhost (my only collector) and view the Performance tab, the "Cycle Times" values for zenperfsnmp for the past 36 hours are as follows:
cur 82.07
avg: 73.35
max: 127.56

So if the graphs are showing cycle times far less than 243.93...then why am I receiving this alert?

Also if I go to /Monitors/localhost and view the Edit tab, my "SNMP Performance Cycle Interval (secs)" is set to 300. So if I'm not exceeding this value...why am I being alerted about it?

Also, it seems very strange that the "current value" is always the same number until a restart....as the count on that alert keeps increasing. If I go through my history, I can see the same alert with slightly different values....but the times they change seem to coincide with a restart of zenoss. For instance, I have on of these alerts that has a count of 173...and another that has a count of 143. I find it hard to believe that zenperfsnmp took 244.57 seconds EXACTLY 173 times in a row.

    What is your zenperfsnmp datapoint rate?

    I have ~260 datapoint rate with a max cycle time of 60...

    So my guess would be that its one particular device thats giving zenperfsnmp trouble. I've seen this happen with my BGP routers: zenperfsnmp was trying to pull the entire routing table off of them, hence the timeouts... so I just turned off the RouteMap collector plugin on those devices, and that fixed it for me.
