Zenoss not alerting on 'quick' open/close events?

Hi
I've got zenoss setup to receive SNMP traps from a number of APC UPS'. I've spent a number of hours setting up classes of events to match all the various open/close, pass/fail conditions that these UPS' send traps on (on battery, off batter, contact open/close, etc). The way I've done this, for example, is to create a class for each pair of triggers, ie 'DoorContacts', which contains the two EventClassKeys that represent the two traps that are sent for contactOpen and contactClose. I've set contactClose to be of clear status and it seems to work as expected...events are moved to the history as expected. I've setup an email alert to send on any event matching the base UPS class (into which all the other DoorContacts, Battery, etc. classes are subs of) and this seems to work fine, when there is at least 10 seconds between the correlated events. It seems that if the two events fire in less than about 10 seconds, they do not get alerted on, but are still put into the proper USP/subclass and moved to the history.

Is there some suppression mechanism taking effect here? I'd like to have it alert no matter what (since, for example, 99% of the time, when a door opens and closes it takes less than 10 seconds and we'd never catch the contact open/close trap if it was moved to the history and never alerted on).

Thanks again for all your help with this great software!

Cheers

J

    Really, what I need to know is if Zenoss does have some function for
    suppression of msgs that fire quickly. If it does not, then the VM
    idea might have some weight...but if Zenoss IS holding the msgs on
    purpose, then it is just something I need to somehow tune.


    While Zenoss doesn't have suppression for events that occur and clear
    very quickly, these kinds of events do go unnoticed by the alerting
    rules quite often. The reason is that the zenactions daemon only
    matches the active events against your configured alerting rules once
    per minute. If an event isn't in the active (status) table at this
    time, it won't be alerted on.

    Running in a VM should have no affect on this at all unless the VM is
    so slow and overburdened that it causes zenactions to run even less
    than once per minute.
