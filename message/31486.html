cronjob-weekly: zeopack fails... a shell problem perhaps?

Greetings...

I'm currently running version 2.4.1

While looking for something else (of course), I saw system mails from CRON indicating the weekly job had failed. I tracked that down to /etc/cron.weekly/zenoss. After some debugging and trying to run it manually as the Zenoss user, it looks to me like the last line of that cron file was meant to run a python script -- but it produced a bunch of bash shell errors.

when I changed the last line of /etc/cron.weekly/zenoss to hardcode the path to the python interpreter, it seemed to run (I think... it's still running now where it had failed immediately previously)

/usr/local/zenoss/python/bin/python $ZOPEHOME/bin/zeopack.py -p $PORT


and.... it finished. My goodness, my Data.fs file went from 3Gigs to 100Megs.

OK -- if nobody else fixed this first, could you send it along as a patch?

Thanks...

--Jay

    RE: cronjob-weekly: zeopack fails... a shell problem perhaps?
    I ran into this to this too, running 2.4.2 on debian using the .deb stack installer. Cron was spitting out:
    /etc/cron.weekly/zenoss:
grep: /etc/zeo.conf: No such file or directory
/etc/cron.weekly/zenoss: line 43: /bin/zeopack.py: No such file or directory
run-parts: /etc/cron.weekly/zenoss exited with return code 127


    After some debugging I found that the cause was that the zenoss environment variables was loaded through ~zenoss/.bashrc which on the first non-comment line returns unless the running shell is a interactive one.
    As .bashrc is returned, the call to load /usr/local/zenoss/scripts/setenv.sh will never be reached causing the $ZENHOME variable to remain unset and the files mentioned in the error message not to be found.

    # If not running interactively, don't do anything
[ -z "$PS1" ] && return

    (from ~zenoss/.bashrc)

    # location of the shell script that initializes the zenoss environment
ZENOSS_ENV=~zenoss/.bashrc

    (from /etc/cron.weekly/zenoss and /etc/cron.daily/zenoss)

    The error will also be apparent when running /etc/cron.daily/zenoss would the last line be uncommented since its the only line in that script that referes to the $ZENHOME variable:
    # run zenbackup to dump the events database as well as zeodb
#${ZENHOME}/bin/zenbackup -C ${ZENHOME}/etc/zenbackup.conf < /dev/null


    The solution I will run, and that I'd propose to at least the .deb distribution would be to hardcode ZENOSS_ENV to /usr/local/zenoss/scripts/setenv.sh since the installer location in the .deb is fixed to /usr/local/zenoss and all the relevant environment vars seems to be in setenv.sh anyway.
    In other words, replace:
    # location of the shell script that initializes the zenoss environment
ZENOSS_ENV=~zenoss/.bashrc

    in /etc/cron.weekly/zenoss and /etc/cron.daily/zenoss with:
    # location of the shell script that initializes the zenoss environment
ZENOSS_ENV=/usr/local/zenoss/scripts/setenv.sh


    It would be really nice to see a fix for this in the official .deb distribution, whether the fix is this or if somebody else have a better and/or more dynamic solution?

    I didn't experience this issue before 2.4.3 but well, i did and just for the record I solved it by prefixing calls to zeopack.py in cron.weekly/zenoss like this:

    ${PYTHONHOME}/bin/python $ZOPEHOME/bin/zeopack.py -h localhost -p ${PORT}


    Please note that cron.weekly/zenoss and cron.daily/zenoss will probably get overwritten when you update your zenoss-install using apt.
