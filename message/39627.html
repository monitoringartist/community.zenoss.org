    Re: still got this one last issue...

    That debug output tells me that the user authentication is failing, for whatever reason. Can you go to the Event Log on a server you are testing with and see what the Security log says? You should see an immediate audit log entry after you try and connect with wmic. Given that output I would expect a failure actually authenticating that user. That would be from a bad username, password, or if the user has been denied access to logon to the machine remotely via group policy. There's probably others I don't know about

     

    If you see a success audit message there, then we're actually hitting the DCOM subsystem and getting rejected there, but I would suspect different debug output from your test in that case. Regardless, try enabling DCOM logging as described at http://support.microsoft.com/kb/892500 - you'll unfortunately have to reboot the server to pick up that change (there might be a service or two to restart that'll do it, but I'm not sure which ones). After that, if you try your test you should see DCOM Error events in the System event log on that box after each connection attempt.

    Re: still got this one last issue...

    Ok, this issue (amongst the many I have) is now offically closed...  Finally!

     

    So the low down is, after spending an hour with MS support, they were able to determine that my user did not have access to the server due to the local users group that had been removed from "acces this computer from the network" in User Rights Assignment of the Local Group Policy.  It was the same for my other servers as well.  This should not normally be the case, which is why 8 out of 300 odd servers weren't working.  This right obviously trumps the COM permissions for Component Services and which is also why nothing was showing up in the event logs.

     

    I did get this afterwards in the SYSTEM event log...  which proves that your domain user needs the Remote Activation permission inthe Component Services to access WMI CIMV2 namespace

     

    The machine wide limit settings do not grant Remote Activation permission for COM Server applications to the user EMEAAD\zenwmi SID (S-1-5-21-1248577188-10479689-3873521419-96259).  This security permission can be modified using the Component Services administrative tool.

     

    The funny thing is, before adding back the local users group, I could logon to the server with my user and run a WBEMTEST connection fine - that is what was screwing with me... anyway it was right in front of my nose.

     

    So, if your company is tight on security and has removed the local users group from "acces this computer from the network" in User Rights Assignment of the Local Group Policy or your domain user in not a member of Domain users, you will need to allow them to this right in the local GPO before WMI access is granted to the server.

     

    It it helps..
