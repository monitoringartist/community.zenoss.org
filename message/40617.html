    Re: development environment for zenoss

    I don't know about Eclipse; I just use vim.

     

    Debugging involves nothing more than dropping breakpoints in the code:

     

    import pdb; pdb.set_trace()

     

    Then running Zope, or zenhub, or whatever relevant process in the foreground:

     

    $ zopectl fg

    $ zenhub run -v10

    $ zencommand run -v10

    ...

     

    Then stepping through it the usual pdb way.

     

    You'll get much more information out of Zope if you turn on debug mode, by adding to $ZENHOME/etc/zope.conf:

     

    debug-mode on

     

    I've never used an IDE that didn't just end up getting in the way more often than it helped.

     

    --Ian

    Re: development environment for zenoss

    As with Ian's experience, although I haven't used an IDE since Turbo C, I find that a few xterms, a couple of instances of gvim, and zendmd do me fine.  Here's what I'm doing lately:

     

    1. I make heavy use of test-driven development, simultaneously editing my Unit Tests in one gvim session, and my code in another gvim session right next to it.
    2. As much as possible, I first develop the code that does not depend on any other zope or zenoss components first, and I bury any such module imports so they don't get pulled in until I need them.
    3. When it's time to talk to zope, I either:
      1. Make appropriate CIFS/NFS mounts so that I can edit the code on my workstation, and run it on a test Zenoss box, or
      2. write some ad-hoc shell scripts to scp the code over to the zenoss box for testing, or sometimes I even
      3. Do an svn commit on my workstation and an svn update on the target zenoss box, but that's ugly.
      4. If I used Linux as my main UI, I'd be developing directly on a zenoss test system, but I use cygwin/X under Windows.
    4. I make frequent use of zendmd to test ideas.
    5. In my unit test script, I pull in much of the framework that zendmd uses, so I can do pretty much anything there; I import pdb into my test scripts so that I can run them interactively for debugging.
    6. I make good use of the logging package, even going as far as adding my own logging levels below level 10 (DEBUG) that I call things like TRACE and SUPERTRACE
    7. Don't forget that, especially if you're developing for Enterprise with it's remote collectors, you may need to commit() between certain types of transactions, and not just at the end of your run.

     

    Make things easy on yourself and figre out how to get it working without an IDE first; then you'll at least know where the problem is if you insist on using and IDE.

     

    Oh, and if you're using ZenScriptBase, and also parsing your own command-line arguments, be sure to clear sys.argv[1:] before you grab a DMD, or because ZenScriptBase will parse them again (unless you want to pass them through intentionally).
