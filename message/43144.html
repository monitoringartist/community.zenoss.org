    The large collector is local to the main server (and to me physically), but it probably would not matter.  The devices it monitors are all over various states.

     

    For RRD's, recall that I split them into different file systems that represent different physical disks.  You can add up the 1-k blocks if you wish:

    107336  /usr/local/zenoss/zenoss/perf
    1341824 /perf1
    1539776 /perf2
    1238244 /perf3
    1276428 /perf4

     

    It looks like about 5 GB total.  It's not alot of data--the RRD files are fixed length at only 35,432 bytes (35k).
    Keep in mind, this is just off the "larger" collector, so it's the RRD data for 1410 devices.  RRD tables are stored on the server that collects them.

     

    Not sure what our largest router is, but I found a larger one that has 991 interfaces, each with 6 polling points (in octets, out octets, in errors, etc)
       so that is 6x991 .rrd files, just for the larger routers.

     

    A smaller router may only have 8 interfaces or so--ethernet, serial, T1, maybe a multilink or VLAN or something.

    Also,

     

    I was retaining data for 12 months, and the MySQL DB grew to about 60 GB on the file system (not in the DB).  This would depend on how many events come in and polling errors you get.  We also monitor for T1 timing slips, and QoS errors throughout our network.  Because of that, I would guess we have a larger-than-normal event history for someone with 1500 devices or so.  However, if you have some special polling or other error generating devices, it could be worse.  I am planning on taking us down to about 90 days of history though.  We just don't have the need to retain it.  I tend to get rid of data quicker than the average admin--unless of course the business requires.

    We backup the DB and the ZOPE DB nightly, I also run the zenmodeler on all servers nightly.  Also pack the ZOPE DB nightly.  This can all happen while Zenoss continues to run.

     

    Here is what the Data.fs files look like.  This is the ZOPE DB, the .fs file is the data store:
    -rw-r--r-- 1 zenoss zenoss 25628973 2009-11-22 19:20 Data.fs

     

    Again, not too big.  You'll want to account for growth.  Also, when you do a zeopack, it builds a new one, so you need more than double the space.

    Here is a quick "show table status", this would be for all events from all collectors.  There is only 1 mysql DB

    Our events go back to 2009-06-15, so that is just under 6 months of data.  This is data rows only, and I am sure there is some space for indexes and such.

    We use the zenbackup command to backup data:
    /usr/local/zenoss/zenoss/bin/

    zenbackup --save-mysql-access --temp-dir=/usr/local/zenoss/zenoss/tmp

     

    This gets the MySql events database, the etc and perf directories, zenpacks, I think the binaries and the zope DB.

    I then pack the zeo DB, and then kick off the zenmodeler.
    Our backup system then picks up the .tgz file and gets us another off-server copy.  Here is a quick look at what is in the .tgz file

    Re: New 36G, 16CPU server

    Thats the design of innodb (transactional tableformat) in mysql.

     

    Its datafile(s) will only grow - not shrink.

     

    Only way to shrink them is to take a textbased backup of the database, delete datafile (along with restarting mysql etc) and then restore the textfile backup.

     

    If you run "OPTIMIZE TABLE" on innodb it will only have the same effect as "ANALYZE TABLE" and that is to sort the indexes.

     

    One way to fix this is to switch to using myisam instead of innodb.

     

    The downside with myisam is that its not transactional (in really short terms has worser performance than innodb with many concurrent clients, however if the client count (accessing the db) is low then myisam performs better).

     

    The good thing with myisam (in this case) is that it fully supports OPTIMIZE TABLE which means that not only index will be sorted but any "holes" in the datafile will be removed since the datafile is completely rewritten (however you should shutdown the zenoss collecting services when you do this because queries will get in a queue until its full and an OPTIMIZE TABLE on a multigigabyte file will take time even if you have fast disks).
