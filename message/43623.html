    Re: Need a hand deleting an OsProcess Instance

    Having had a bit more of a dig around, I'm not convinced that the hex numbers are to do with events. I think they may be more to do with Views that show details of a process instance for a device.  Try navigating down $ZENHOME/perf/Devices/<device name>/os/processes . I have subdirectories roughly of the format:

    <proc name> <hex number>

    .. though some of them don't have the hex number.  Under these directories are the RRD files for count, cpu and memory for the process instance.

     

    You can also see this from the GUI - Navigate to /Processes and go down into a specific process. Click on the Name link for a process instance and the title bar of the resulting view  shows that same hex number.

     

    I suspect our problems come when somehow this View gets corrupted.

     

    Cheers,

    Jane

    Re: Need a hand deleting an OsProcess Instance

    I have now managed to cleanup our process definition inconsistencies.  We have process monitoring for the Zenoss processes themselves under the suborganizer Zenoss.  The breadcrumb /Processes/Zenoss showed all the processes but clicking on the broken ones gave an error.  Another symptom is if the count of the process instances is higher than it should be.  In our case, we had counts of 3 against zenwin, zeneventlog, zenprocess, zencommand and zenhub.  For all except zenhub, the count should be 1 (as there is only one of each daemon running and they only run on a single server).  zenhub should have a count of 2 as we use zenhub workers so the regex of zenhub matches zenhub and zenhubworker.

     

    If these broken processes were examined under the OS tab of a specific device, then everything is OK - you can see all the data and it is being actively collected (once we had used zendmd to do a reindex() and a commit() ).  You just can't see / change the definition of the process.

     

    I followed rbilder's advice to cleanup the processes, which worked for all the processes except zenhub.  That one, I finally had to delete the definition of the zenhub process under the /Processes breadcrumb and then run reindex() and commit() under zendmd.  I could then add the definition back, remodel the device and it not only picks up the definition but I can also see all the old data for zenhub as well as collecting new.

     

    Incidentally, the other piece of Python I have found useful in zendmd to see what devices have what instances of what processes is:

     

    for d in dmd.Devices.Server.getSubDevices():

        for p in d.os.processes():

            print d, p.id, p.procName, p.parameters

     

    The only glitch I think I have remaining is that  $ZENHOME/perf/Devices/<host name/os/Processes has some old subdirectories containing redundant RRD files.  I thought there was a mechanism to clean these up???? but they still seem to be there.  At least they are not causing pain.

     

    Hope that helps someone else.

    Cheers,

    Jane
