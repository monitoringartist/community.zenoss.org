    Re: Buffering alerts, so that we receive one email alert per minute maximum

    I wouldn't quite call it an epiphany! 

    But I have improved the situation quite a lot.

     

    In this ticket I was asking for a way to set a maximum number of emails per minute for any particular rule.  I also wanted the emails to contain information for multiple events (so although you're possibly getting fewer emails in total, you get all the same information).  This is because I've had the experience of a process going haywire and generating lots of syslogs very quickly, all with slightly different message summarys (maybe there's an increasing number in there or something) so each and every one becomes an event for which an email is sent.

     

    A friendly Zenoss forum user showed me how to get at the Zope interface at some point - you can use it to get closer to the SQL queries behind your alerting rules (rather than having to use the Zenoss GUI).  Go to the page for your particular rule in Zenoss and append '/manage' to the URL.  In the new page, click 'properties'.  You now have a box where you can define your own SQL where clause!  Once you've done that, click save changes and bear in mind that if you subsequently edit the rule using the Zenoss GUI you will overwrite what you have done.

     

    You can become familiar with the tables and fields by jumping onto the mysql DB directly - on your Zenoss box as the Zenoss user, type 'mysql -u zenoss -pzenoss' - unless you changed the default password, that should work.

     

    The trick is to use the where clause to contain your own query.  For us, the process name is stored as the 'component' field, so I think I used something like this (I'm not at work at the moment so can't remember exactly!): -

     

    WHERE : (eventState=0) and (productionState = 1000) and (agent = zensyslog) and component not in

                                                              (select component from status inner join alert_state on status.evid=alert_state.evid where rule = 'email_rule')

     

    An event only gets an entry in the alert_state table when a rule has alerted on it.  So by doing an inner join with the status table and forcing the rule name to be the relevant email rule in question, the sub-query is returning the list of processes (components) that have an active event in the status table for which an email has already been sent.  By saying in the main query that I don't want the process (component) that's generated the event in question to be in that list, what I'm saying overall is this: -

     

    If the process that's generated the current event has previously generated an event that hasn't been closed yet and for which we've already been alerted, don't bother to send another email about it (please). 

     

    It's still not perfect.  The zenAction cycle time is by standard 1 minute, and I'm not sure how big an impact reducing this has, so I've not fiddled with it.  But I've found that on each cycle zenAction seems to process all events simultaneously.  So say a process goes beserk and generates an event every second.  We're half way through the zenAction cycle, so from a clean status table I get 30 new events for the same component.  When zenActions is run, it sends 30 emails because it's not previously alerted on the process before.  After that cycle, though, it will send no more until you clear the events (or in particular, you clear the particular event for which the email was sent - if you clear that event on it's own, then on the next cycle you'll get emails for all the rest, so be sure to clear them all at the same time!).  So the new maximum number of emails for a process is however many events it can generate in one zenAction cycle time!

     

    A really good addition would be to use the 'limit' funtion - something like 'where evid in (select evid from status where etc. etc. limit 10)'.  You'd arbitrarily be able to set a maximun number of emails.  I've found that although the command works on the mysql db directly, Zenoss (ie. zenActions log) complains that mysql server doesn't support it.  I don't know if you can just update the mysql server version?
