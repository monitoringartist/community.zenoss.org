    Re: non-latin chars in Eventlog

    I think  I solved the problem. The root of the problem is incorrect encoding of the events database. Correct settings should be:


    mysql> show variables like 'coll%';
    +----------------------+-----------------+
    | Variable_name        | Value           |
    +----------------------+-----------------+
    | collation_connection | utf8_general_ci |
    | collation_database   | utf8_unicode_ci |
    | collation_server     | utf8_unicode_ci |
    +----------------------+-----------------+
    3 rows in set (0.00 sec)

     

    Below is step-by-step recipe how to enable Zenoss 2.5.2 for working with non-latin chars.

     

    1. Patch file $ZENHOME/Products/ZenUtils/json.py. Change encoding from iso-8859-1 to utf-8. That's strange, but 2.5.2 is shipped with 'latin-1' hardcoded

     

    def inner(*args, **kwargs):
                return simplejson.dumps(value(*args, **kwargs),
                                        encoding='utf-8' ) #iso-8859-1
            # Well-behaved decorators look like the decorated function
            inner.__name__ = value.__name__
            inner.__dict__.update(value.__dict__)
            inner.__doc__ = value.__doc__
            return inner
        else:
            # Simply serialize the value passed
            return simplejson.dumps(value, encoding='utf-8') #iso-8859-    1

     

    2. Shutdown zenoss

    3. Change mysql config. In my installation config is in the file  /usr/local/zenoss/mysql/my.cnf.

    Two section in the config should be appended with encoding instructions:

     

    [mysqld]
    basedir=/usr/local/zenoss/mysql
    datadir=/usr/local/zenoss/mysql/data
    port=3307
    socket=/usr/local/zenoss/mysql/tmp/mysql.sock
    tmpdir=/usr/local/zenoss/mysql/tmp
    default-character-set=utf8
    collation_server=utf8_unicode_ci

     

    [client]
    port=3307
    socket=/usr/local/zenoss/mysql/tmp/mysql.sock
    default-character-set=utf8

     

    4. Restart mysql (e.g. by /usr/local/zenoss/ctlscript.sh restart mysql)

     

    5. Drop events database

     

    mysql -u zenoss -p events

    >>drop database events;

    >>exit;

     

    6. And at last the most tricky task: Create new events database with correct coding. I'm not a guru in zenoss - I didn't find a regular database creation script, so I made my own script on the base of $ZENHOME/bin/zenoss_init_pre. Just copy zenoss_init_pre to whatever file you want and edit this file, comments out all tasks exept create_database and run_zeneventbuild. The result should be like below. Then run the file (under root!) and check results by connecting to mysql and issuing commands: show databases; (shluld be database events) show variables like ' coll%' (encoding should be utf-8).

     

    ...

    # set the python shebang line

    shebang

     

    # create the log directory

    ##mkdir -p ${ZENHOME}/log

     

    # restart external server daemons

    ##restart_snmpd

    ##restart_mysql

     

    # copy the etc/*.conf.example files to etc/*.conf if etc/*.conf don't

    # already exist (we don't want to clobber people's configs)

    ##update_conf_files

     

    # create the events mysql database

    create_database

     

    # populate the events database

    run_zeneventbuild

     

    # set up the zope instance

    ##run_mkzopeinstance

     

    # create the zeodb instance

    ##run_mkzeoinstance

    ...

     

    7. Start zenoss and enjoj! In my case I can see events with russian messages (which come from my Windows-servers), I can write and add comments in russian to the event and I got rid of 'Server connection error' problem.

     

    Updated: Sorry, I was wrong: "Server connection error" still occurs. But, I think, I have an  idea on workaround.

     

    Updated: In the attachment is the staff which workarounds  the problem "Server connection error" for my Zenoss 2.5.2 installation.

    Re: non-latin chars in Eventlog

    Hello, James,

     

    I was wrong, database coding has noting to do with the problem "not dispalying not latin chars". Today I made a "clean" experiment. I've installed a fresh 2.5.2 (from the scratch). Then I tried in the Event Console add a log message (in Russian, utf-8) to event record. It was sucessful! The only thing that was wrong: the message in the event-detail-window was displayed in wrong coding. After then I changed coding in the json.py (see my post above). Log message immediatly began to be displayed in correct way. So database has nothing to do with non-display of non latin-1 chars, since database still was in shipped coding.

     

    The discussed issue has a linked one. After I've enabled non latin chars in event console, then in a short period of normal running, Event Console jammed, reporting "Server connection error". In my case (in my installation) I already know why this happens. My Windows WSUS server two times in a day sends a message saying that something is wrong with clients (this mesage is in Russian, windows-1251 encoded). When Event Console is going to display this message, it triggers jason(value) for raising UnicodeDecodeError exception. Zenoss 3.0.0 has the this issue as well. You will face "Server connection error" when will try to open detaile window for such erroneous record.

     

    So, the problem obviously has a deeper roots. Unfortunatly, my knowleadge in both python and Zenoss unsufficent for such drill down. It's a pitty. But, if needed, I could make a dump of events database with erroneous record. Maybe this data will help developers to find out the real cause of the problem.

    Re: non-latin chars in Eventlog

    I think I found the solution. my zenoss is 3.0.2, my unichar is Simple Chinese.

    the file 'eventfacade.py' in  $ZENHOME/Products/Zuul/facades/ fills the 'summary' field's value with decode('iso-8859-1'), I changed it to 'utf-8', it works.

     

    here's what I have done.

     

    file: eventfacade.py

    file path:  $ZENHOME/Products/Zuul/facades/

    before change: (line 152 ~ 154)

              if field not in data:
                    if isinstance(value, str):
                        value = value.decode('iso-8859-1')

    after change:

              if field not in data:
                    if isinstance(value, str):
                        value = value.decode('utf-8')
