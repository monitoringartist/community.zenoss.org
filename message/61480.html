    Re: Redefining a class

    You can change a class' security definitions without overriding the whole

    class. Assuming you want to override the security for the Device.setManageIp

    method...

     

    import Globals

    from AccessControl import ClassSecurityInfo

    from Products.ZenModel.Device import Device

    from Products.ZenModel.ZenossSecurity import ZEN_VIEW

    Device.security = ClassSecurityInfo()

    Device.security.declareProtected(ZEN_VIEW, 'setManageIp')

    Globals.InitializeClass(Device)

    Based on the similarity that probably did come indirectly from me. I see

    what you mean about RenderServer and ZenossPortlets.

     

    The reason you can't set the ZEN_COMMON permissions for RenderServer.render

    is that it already has a security declaration. Zope won't allow you to

    change existing class security definitions. The easiest way I've found to

    get around this is to allow anyone with the "Authenticated" role access to

    RenderServer.

     

    The reason you're having trouble with ZenossPortlets is that the portlet

    permissions have to be changed, then they have to be re-registered.

     

    You can find an example ZenPack __init__.py at the following URL that

    demonstrates both of these approaches.

    Re: Redefining a class

    I have no idea how I put that code into the example

    "allowAuthenticatedRender" method. It's totally wrong. I've updated the code

    accordingly. What I wanted to do was add the ZEN_VIEW permission to the

    'Authenticated' role on the RenderServer object. Check the gist again.

     

    It's true that Zope does do a bit of work to prevent you from changing

    security declarations that have already been made. This means that if you're

    trying to change the security declaration of a method that doesn't already

    have one, you're fine. Otherwise you're not. There probably is a way to

    override the existing security, but after about 20 minutes of searching I

    wasn't able to find out what it was so I took a different approach.

     

    Other answers..

     

       1. The logging module is just the standard Python logging module. Search

       for "python logging" and you can see the extensive range of options

       available. In the case of ZenPack install routines I'd recommend just

       leaving it as it is in my example as this output is usually best off in the

       console and redirected elsewhere with the shell if desired.

     

       2. The reason you'll see "import Globals" in a lot of Zenoss code is that

       it sets up the PYTHONPATH appropriately to make sure all of the needed

       modules are available to your code. Doing "from Globals import *" will do

       the same thing, but it will also get you access to all of the methods in the

       Global module like InitializeClass. So if you just did "import Globals"

       you'd need to do Globals.InitializeClass(...) whereas with "from Globals

       import *" you could just do InitializeClass(...).

     

       3. Importing specific classes and methods from modules is often regarded

       as a best practice as it is more explicit. This holds true across languages

       beyond just Python. The main reason I do it is that I use a pyflakes

       extension to vim that lets me know when I've imported something and not used

       it, or used something that I haven't imported. It works much better with

       explicit imports.

     

       4. There is no longer any reason to create an update method in a ZenPack.

       This is deprecated. These days (starting with 2.5 I believe) when you

       upgrade a ZenPack it first called the remove method with the leaveObjects

       parameter set to True, then calls the install method.

     

       5. Using super instead of statically calling the super-class method is

       slightly more maintainable in that if you change the class your class

       inherits from you will have to go back through and update all of your static

       references to it. Personally I like the syntax better. Be aware that if your

       class is inheriting from multiple classes that super will invoke the method

       on the first super-class. So if you need to call a super-class method on the

       non-primary super-class you should still use the static type of reference.
