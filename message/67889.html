    Few suggestions for you to try.

     

    In order to make sure zenoss user has been created, at the server console or through a SS session, run the following command :    cat /etc/passwd | grep "zenoss". If you don't get any reply, zenoss user has not been created and you may force its creation running as root user the following command    useradd -m zenoss.

    Other suggestion, setup sudo so you can be able to run    sudo su - zenoss   using your password instead of zenoss password.

     

    Hope that will help

    Re: cannot start the new zenoss on cerntos 6.3

    You are right, it is not a good idea to reset the password for zenoss account. What I'm doing is using sudo to allow me to just type  sudo su - zenoss  and be logged in as user zenoss after being prompted for my own password. Read the man pages for sudo and sudoers and use visudo to modify the /etc/sudoers file (it should be something like adding the line   your_user ALL = /bin/su - zenoss ).

    As for starting the zenoss service or any other service, you should do this as root or using the root password with su and certainly not under a regular user account as is the case for zenoss. By default, only root has the necessary privileges to start/stop services.

    I've just installed Zenoss Core v4.2 on CentOS v6.3 manually (using rpm not the script) following the documentation and apart from some minor modifications related to Java and MySQL installation, everything went ok. I can start zenoss service as root witho no error message.

    If you want to dig into zenoss init scripts on CentOS I would suggest you to start with /etc/init.d/zenoss

    Re: cannot start the new zenoss on cerntos 6.3

    Hi Nillie and everyone who replied

     

     

       I was able to solve the issue

     

     

    With centos 6 and up, they give you a number of configurations to load   workstation, server, etc....   I went into the code for the different configuration installs and after reviewing the zenoss script, the manual install, and some of the CentOS source code   found that the Centos setup was the issue as i did the install as a SERVER (security reasons)  and not a WEB SERVER.  The 2nd item that I had to correct was te selinux. This was an easy fix  by editing the /etc/sysconfig/selinux  file by setting SELINUX=disabled.  It was as simple as that. I hope this will help any others that come accross this type of issue
