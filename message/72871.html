    Re: WARNING zen.zeneventlog: Queue exceeded maximum length: 6372/5000. Trimming

    Just my 2 cents here, I think this is a threshold which is being exceeded, but whats making the  event queue grow beyond the set threshold? My belief is that when zeneventlog loses its heartbit, it stops processing events. This means that events will start to pile up. At that point, intervention is needed, either to clear the heart beats or to restart this particular daemon.

    So this takes up further back to the question, "what causes a daemon to lose its heartbeat?" This is where I believe there is a lot of bone of contention. One can simply say that its to do with the perfomance of the server (workers, cycle times, load, timeout settings etc) Configuration issues cannot be ruled out. Perusing the zeneventlog.log may give some insight. It might be the daemon or the hub hits some specific condition which boots the daemon out beat.

    Re: WARNING zen.zeneventlog: Queue exceeded maximum length: 6372/5000. Trimming

    This question about maxqueue length has been asked before and I have never seen an answer by any zenoss expert.

    I personally believe the maxqueue length is a safety limit in the daemon, if  something were to happen that produced tons of events, there would be some throttling of them. If not the event would overwhelm zenhub, causing heartbeat errors. But with the maxqueuelenth some events are dropped but the alternative would be all events would be dropped. THis is just my 2 cents not based on any hard data. I have in the past just bumped up the value of the maxqueuelength will no ill effects, but I would bet there is a upper limit.
