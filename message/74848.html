    Jane,

     

    I do apologize for the extreme tardiness, there is no excuse for it. The good news is we now have more PM bandwidth to share the load.

    A couple of things that are coming out in the next release (4.2.5 - slated January 31st 2014):

    * Enhanced process monitoring - support for varabile CLI args vs. static MD5 hash function today which does not support process starts with different CLI args (this is a major upgrade)

    * Event Flap Detection - Yes, finally it's here

    * Enhanced event console filtering - regex/reverse filtering supported again amongst other things that were dropped between 3.2.1 and 4.2 (due to new Lucene index implementation)

    * Numerous UI improvements which several of them are in 4.2.4 including better event transform workflows and context/code syntax highlighting amongst other things

    Re: Suggested Features

    Below is a break down of each request and a response. Our next major release is slated for mid next year with the expectation that we will ship an alpah towards the end of this year/early next year. More details will be forthcoming over the next couple of months.

     

      1. Ability to monitor processes and services for device classes (not just a global default and a specific device instance)

     

    Min/Max threshold can now be set for process classes (not just specific devices/components) -> 4.2.5 release

     

      2. Automation to move devices from /Discovered to more helpful device classes - see http://community.zenoss.org/docs/DOC-3540

     

    In backlog (generic & mapping mechanism). We have some of this support today for customers using our CiscoMonitor ZenPack but it isn't holistic

     

      3. Ability to export graphs and reports - to separate webserver and to pdf files.  Fix the "export all" button.

     

    Export graph (to CSV) functionality will be present in Europa release (next major Zenoss release), will be supported on a per graph basis

     

      4. Good documentation on ZenPack design

     

    Ongoing and will be published at will. ZenPack Generator should go along way towards addressing this requirement


      5. Good documentation on new architecture in Zenoss 4 - event subsystem, Rabbit, Zope database in MySQL,..

     

    In backlog, we have improved user documentation for the 4.2.5 release but it's going to change somewhat for Europa


      6.

        1. Ability to schedule command checks, (ie run once daily @10am)

     

    In 5.X backlog, good enhancement idea


        2. Custom properties for components as well as devices, (e.g. to set thresholds per-component)

     

    In 5.X backlog, will require some refactoring of the back-end to support effectively (non trivial to manage)

     

        3. Ability to define multiple rules for an event view rather than just one big rule

     

    Not sure I understand this one. Event filtering will change dramatically in 4.2.5 with full regex support and a basic SQL like DSL implementation for querying events. Applying multiple rules should be possible via SQL/DSL implementation.

        4. Set multiple thresholds and only event for the highest breached threshold, (so we don't get a CPU warning *and* CPU critical event when the critical threhold is breached)

     

    In backlog, interesting noise suppression concept, would like to see this developed as an (pre/pro) event plug-in.

     

    Scheduling zenmodeler runs - not just frequency.

     

    Is the rationale to update the model more regularly i.e. for certain targets run @ 10am knowing they will be turned on vs. @ 10PM when they will be off? We use event based triggers to drive modeling today but on specific events. Have you looking at this approach? i.e. generate a specific event at a certain time and use that to drive a remodel?

      7. - Easy device dependacy to create a event suppression logic (to lower alarm for remote monitoring over ipsec)

     

    We have support for this in Service Impact (commercial offering), we would like make event-plug-ins simpler that can apply to existing dependencies contained within Zenoss (backlog)

      8.

    - an option to select time range in reportmail export.

    In backlog

     

    - a full integration and optimization of phpweathermap

    Probably not going to happen unless everyone asks for it

     

    1) A built-in MIB Browser, a built-in MIB Browser, a built-in MIB Browser  - not just a ZenPack that I have to fix with evry new release!  PLEASE!

    In backlog, good idea

      

    1: Run commands against components.

    In backlog, good idea though will require further though since not all components can be acted on directly or without additional context. The Windows use case is straight forward since you can use the service controller to start/stop services by host/service name.

     

    It seems that the commands menu in a device (and in the events window) only seems to run against the device, not the component selected in the sub-window or the component noted in the event.

     

    So if I had a command to start, stop or restart a windows service I would need a command for each service I can predict restarting rather than one command which would take device/component or evt/component as the service I wish to restart.

     

    2: A context-sensitive action (cog-wheel option) for the Infastructure side window that lets me run a remodel against an entire device-class, group, system or location.

    In backlog 


    3. Ability to link network ranges to a location (I think some preliminary work has already been done in Zenoss for this) and have the regular modeling adjust the location for the device depending on it's IP. Probably need a setting against devices to allow prevention of automated location changing based on IP).

    In backlog, will require support for IPv6.  


    Great if you're monitoring Laptops and devices by host names which might xenmotion/vmotion between sites and pick up new IP addresses.

     

    4. Not so much a feature but it'd be nice to have a lower, alternate licensing price for user devices (e.g. desktops) as opposed to servers.  Likely some definition required around it to specify systems providing for only a single user or a single user at a time (workstations) and not using extended monitoring or something.  Lawyers can figure it out but it could be good for sites with thousands of small registers in stores or terminals in branch offices and other such devices which are often considered far less vital than the full servers but would add some small value if they were all monitored.

     

    AND/OR a license for modeling devices only. i.e. the ability to discover and model devices for asset registration, an up-to-date CMDB and general tracking but no permission to run any monitoring template (even pings) against the device.  Possibly need to restrict frequency of modeling.

     

    So, I might buy 500 Server Monitoring licenses, 1000 User Monitoring licenses (Head office PC's/Laptops) and 6000 Modeling Licenses (store registers/printers) to cover the business needs.  You might even get some customers who wish to use it as a pure CMDB/asset management system and only buy Modeling licenses.

     

    Interesting use case, will take into consideration when the license model is reviewed next.
